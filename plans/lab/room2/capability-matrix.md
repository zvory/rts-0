# Room Capability Matrix

This is a current-state baseline for the room capability cleanup. It records what the code does
before protocol names, client affordances, or runtime behavior are changed in later phases.

## Product Path Baseline

| Product path | State source and lifecycle | Clock | Authority and mutation | Visibility and diagnostics | Persistence/export | Start payload and client affordances | Manual smoke focus |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Normal lobby | `RoomTask` lobby state: joined humans, AI seats, spectators, ready state, selected map, and countdown state. Joins use normal lobby admission. | `ClockPolicy::RoomTicker`; scheduled ticks are no-ops unless countdown is active. | Host-owned lobby controls mutate lobby state: teams, factions, AI seats, map, spectator status, quickstart, and start requests. | Lobby messages only; no game snapshot projection and no observer diagnostics. | No match history or replay export yet. | No `start` payload until launch. Client shows lobby, host controls, spectator controls, map selector, and start affordance. | Join as host, add AI, move a player to spectator, change map, ready/start countdown. |
| Normal live match | Authoritative `Game` in `Phase::InGame`; new joins are rejected once the match is live. | `ClockPolicy::LiveMatch` at the configured tick rate. | Active players issue normal gameplay commands through the public `Game` command path. Spectators are read-only. | Active players receive player-fog snapshots. Spectators receive a union of active-player fog and observer analysis. Movement debug paths are not available unless the start payload's `debugMode` is true. | Match history and replay persistence are eligible for normal, non-automated rooms when DB recording is enabled. | `StartPayloadPolicy::LiveMatch`; player starts enable prediction, spectator starts disable prediction. Client settings use match or spectator context. | Start a two-player match, issue commands as a player, confirm a pre-match spectator sees union vision and cannot command. |
| Live spectator | Same live `Game` as the normal match; spectator role must be established before launch. | Same live match clock. | Read-only; command surfaces are hidden or disabled by spectator state. | `SnapshotProjection::SpectatorUnion` plus observer analysis for live spectators. | No separate persistence effect beyond the owning match. | Same live start payload with `spectator: true`, prediction disabled, observer overlay mounted for spectators. | Watch a live match as spectator, confirm observer analysis appears and give-up/command controls stay hidden. |
| Post-match replay | A normal room transitions from ended `Game` to `Phase::ReplayViewer` using a captured replay artifact. Replay joins require confirmation. | `ClockPolicy::ReplayPlayback`; speed, pause, seek, and keyframe playback are owned by the replay session. | Replay viewers can change playback speed, seek, select replay vision, and request a replay branch. Gameplay commands are not applied. | Per-viewer `ReplayVision` snapshots use selected player fog or player-union fog. Observer analysis is sent to replay viewers. | No new match-history write for playback; the original normal match may already have persisted a replay artifact. | `StartPayloadPolicy::ReplayViewer` with replay metadata. Client mounts `ReplayViewer`, replay speed/pause/seek, replay vision buttons, timeline, and branch button. | End a match into replay, change speed, pause/resume, seek, switch vision, request a branch. |
| Persisted replay room | `RoomMode::Replay` holds a persisted replay artifact. The room starts in lobby-shaped setup, prompts on join, then transitions to replay playback after confirmation. | Room ticker before the viewer starts; replay playback clock after `Phase::ReplayViewer` is active. | Replay viewer controls only after playback starts. | Replay vision and observer analysis after confirmation. | Playback itself does not write match history. | Replay start payload is sent after confirmed join; client uses replay viewer affordances. | Open a match-history replay room, confirm the prompt, verify playback starts at replay tick zero. |
| Saved replay artifact | `RoomMode::ReplayArtifact` loads a saved self-play artifact from server target artifact directories. Confirmed join starts the shared replay runtime. | Room ticker before load/start; replay playback clock once the viewer is active. | Replay viewer controls only after playback starts. | Replay vision and observer analysis after confirmation. | Playback itself does not write match history. | Replay start payload is sent after confirmed join; client uses replay viewer affordances. | Open a saved artifact replay, verify prompt, playback, vision selection, and timeline marks. |
| Replay branch staging | `RoomMode::ReplayBranch` holds a frozen replay branch seed and `BranchStagingState`. Joins attach to branch staging and are spectator-shaped until launch. | `ClockPolicy::BranchStaging`; no simulation ticks, but countdown can promote a ready branch to live. | Branch occupants claim and release original replay seats; branch host starts launch once every required seat is claimed. | Branch staging messages expose original seats and claims. No game snapshots or observer diagnostics. | Suppressed; staging does not write match history. | No game `start` payload while staging. Client mounts branch staging UI from `branchStaging` messages. | Claim/release seats, verify exclusivity, start only when all original human seats are claimed. |
| Replay branch live | Branch staging promotes to a cloned live `Game` from the replay keyframe. Connected ids are mapped to original replay seat ids. | `ClockPolicy::LiveMatch`. | Claimed occupants issue normal gameplay commands as their original replay seats; unclaimed/extra viewers are spectators. | Active branch seats receive player-fog snapshots for the original seats. Spectators receive active branch-seat union vision. | Suppressed; branch live matches skip public match history. | `StartPayloadPolicy::ReplayBranchLive`; claimed seats get prediction-enabled starts with mapped `playerId`, spectators get spectator starts. | Launch a branch, command as a claimed original seat, verify spectator union vision and no public history write. |
| Dev scenario | `RoomMode::DevScenario` starts a purpose-built dev `Game` through `start_dev_session`; all connections are watchers. | `ClockPolicy::DevWatch`; `setReplaySpeed(0)` pauses, positive `setReplaySpeed` resumes/clamps, `stepDevTick` advances one tick while paused. | Dev-watch controls mutate the dev driver clock and scenario tick progression. Viewer gameplay commands are ignored. | Full-world snapshots via `snapshot_full_for`; dev setup enables runtime movement debug paths. Client `devWatch.kind === "scenario"` reveals all fog and enables all debug paths. | Suppressed; dev-watch end flow does not capture replay artifacts for history. | Normal `start` payload with `spectator: true`, debug mode from the dev `Game`, plus URL-local `devWatch` client affordances. Client shows dev pause/step controls and hides replay seek. | Open `/dev/scenario`, pause with speed 0, step one tick, resume, confirm path diagnostics and no-fog view. |
| Lab operator | `RoomMode::Lab` starts a real room-owned `Game` with lab metadata. First joiner becomes operator; lab session tracks dirty state, vision, and operation log. | `ClockPolicy::LiveMatch`; no special lab time control in this baseline. | Operator sends privileged lab envelopes. Setup mutations call public `Game` lab APIs, `issueCommandAs` routes gameplay commands as a selected player, `setVision` changes lab projection, and export reads scenario state. | Default full-world projection using the lab view player; team/team-union lab vision can switch snapshots to `snapshot_for_spectator`. Observer analysis is not mounted. | Match history suppressed. Scenario export is allowed through lab op results but is not durable public storage. | `StartPayloadPolicy::Lab` with `lab` metadata role `operator`. Client mounts `LabPanel` and lab control policy. | Open `/lab`, spawn/move/delete or edit resources, switch lab vision, export a scenario, issue a command as one player. |
| Lab read-only viewer | Additional lab joiners attach to the existing lab session as read-only viewers and receive the same running lab `Game`. | Same live match clock. | Read-only viewers cannot send lab requests; normal gameplay command surfaces route through lab policy and reject non-operator issue-as. | Same lab projection as the operator, including full-world or selected-team lab vision. Observer analysis is not mounted. | No separate persistence/export authority. | Lab start payload has role `readOnly`; client gets lab state but no operator authority. | Join a lab from a second client, confirm the viewer sees lab state/snapshots but lab mutation requests are rejected. |

## Mode-Shaped Leaks Inventory

| Leak | Current shape | Remove or replace in |
| --- | --- | --- |
| Replay-named time commands | Client and server still use `setReplaySpeed`, `seekReplay`, `seekReplayTo`, and `stepDevTick`; dev scenario pause also rides through replay speed. | Phase 3 |
| Replay-named time state | Server broadcasts `ReplayPlaybackState` for replay playback and dev-watch pause/speed state. | Phase 3 |
| URL-local `devWatch.kind` controls | Client `App` and `Match` use `devWatch.kind === "scenario"` to join, reveal fog, enable debug paths, and mount scenario speed/step controls. | Phase 5 |
| Start payload `debugMode` | `GameState` exposes waypoint diagnostics from `startInfo.debugMode`, while dev scenario also forces availability from `devWatch.kind`. | Phase 4, then Phase 5 |
| `DebugHuman` diagnostic coupling | Movement debug paths are produced by `Game.debug_path_overlays`, which is currently tied to `StartingLoadout::DebugHuman` and dev scenario setup. | Phase 4 |
| Projection diagnostics tied to vision | `ProjectionPolicy::observer_analysis_audience()` infers observer analysis from `VisionPolicy::LiveFog` or `ReplayVision`; diagnostic capability is not independent. | Phase 4 |
| Replay identity in client affordances | `ReplayControls` decides pause, seek, vision, branch, and scenario step controls from `isReplay`, `isScenario`, and `replayViewer` booleans. | Phase 5 |
| Replay branch live joining | `SessionPolicy` still reports `JoinPolicy::BranchStaging` for replay branch live, so branch product identity is doing more than a neutral join capability. | Phase 2 |
| Match-history helper by mode | `SessionPolicy::allows_match_history()` currently checks `SessionMode::Normal` directly instead of the persistence policy alone. | Phase 2 or Phase 6 |
| Lab mutation hidden behind authority | `SessionPolicy` exposes `AuthorityPolicy::LabOperator` and `MutationPolicy::LabReadOnly`, while operator setup mutation, issue-as gameplay, vision changes, and export are separated inside `RoomTask::on_lab_request`. | Phase 2 |
| Lab projection uses replay helper for team vision | Lab selected-team projection passes through `ProjectionPolicy::replay_snapshot_for` because it wants a player-union snapshot, not replay semantics. | Phase 4 |
| Observer overlay mount by product booleans | Client observer analysis mounts from `payload.replay`, `payload.spectator`, and `replayViewer`, not explicit diagnostic metadata. | Phase 5 |

## Characterization Coverage

- `server/src/lobby/session_policy.rs` now has a room2 capability baseline test covering every row above at the `SessionPolicy` level.
- Existing `server/src/lobby/tick_control.rs` tests cover replay playback, replay pause, dev-watch pause/step eligibility, branch staging no-op ticks, and countdown ticks.
- Existing `server/src/lobby/projection.rs` tests cover player fog, branch/live seat aliases, live spectator union, replay vision, dev full-world snapshots, lab full-world snapshots, and observer analysis audience.
- Existing `server/src/lobby/room_task.rs` tests cover lab operator/read-only behavior, lab projection, lab issue-as, dev scenario pause/step, persisted and saved replay rooms, replay vision, replay branches, branch live seat aliases, and branch history suppression.
- Existing `tests/client_contracts.mjs` coverage locks the current client affordances for debug path availability, replay controls, scenario pause/step controls, replay vision, branch button, lab client/panel operations, and observer analysis mounting.
