# Room Capability Matrix

This is the final-state capability map for the room capability cleanup. Product modes still name
entry points and setup flows, but shared room behavior now flows through explicit session,
projection, persistence/export, and start-payload capability policy.

## Product Path Baseline

| Product path | State source and lifecycle | Clock | Authority and mutation | Visibility and diagnostics | Persistence/export | Start payload and client affordances | Manual smoke focus |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Normal lobby | `RoomTask` lobby state: joined humans, AI seats, spectators, ready state, selected map, and countdown state. Joins use normal lobby admission. | `ClockPolicy::RoomTicker`; scheduled ticks are no-ops unless countdown is active. | Host-owned lobby controls mutate lobby state: teams, factions, AI seats, map, spectator status, quickstart, and start requests. | Lobby messages only; no game snapshot projection and no observer diagnostics. | No match history or replay export yet. | No `start` payload until launch. Client shows lobby, host controls, spectator controls, map selector, and start affordance. | Join as host, add AI, move a player to spectator, change map, ready/start countdown. |
| Normal live match | Authoritative `Game` in `Phase::InGame`; new joins are rejected once the match is live. | `ClockPolicy::LiveMatch` at the configured tick rate. | Active players issue normal gameplay commands through `MutationPolicy::LiveGameplayCommands`. Spectators are read-only by participant role. | `VisibilityPolicy::LiveFog`; spectators receive a union of active-player fog and `DiagnosticPolicy::LIVE_SPECTATOR_OBSERVER_ANALYSIS`. Lobby Debug mode adds owner-only movement-path diagnostics for active players through projection policy. | `PersistencePolicy::MATCH_HISTORY_AND_REPLAY_ARTIFACTS`; normal, non-automated rooms are eligible for match-history rows, transient post-match replay capture, and durable match-history replay artifacts when DB recording is enabled. | `StartPayloadPolicy::LiveMatch`; `AffordancePolicy::LiveMatch`; player starts enable prediction and advertise `capabilities.commands.gameplay: true`, spectator starts disable prediction and advertise read-only commands, and diagnostic affordances are advertised in `diagnostics`. | Start a two-player match, issue commands as a player, confirm a pre-match spectator sees union vision and cannot command. |
| Live spectator | Same live `Game` as the normal match; spectator role must be established before launch. | Same live match clock. | Read-only; command surfaces are hidden or disabled by spectator state. | `SnapshotProjection::SpectatorUnion` plus observer analysis for live spectators. | No separate persistence effect beyond the owning match. | Same live start payload with `spectator: true`, prediction disabled, observer overlay mounted for spectators. | Watch a live match as spectator, confirm observer analysis appears and give-up/command controls stay hidden. |
| Post-match replay | A normal room transitions from ended `Game` to `Phase::ReplayViewer` using a captured replay artifact. Replay joins require confirmation. | `ClockPolicy::ReplayPlayback`; speed, pause, seek, and keyframe playback are owned by the replay session. | Replay viewers can change playback speed, seek, select replay vision, and request a replay branch through `MutationPolicy::ReplayPlaybackCursor`. Gameplay commands are not applied. | `VisibilityPolicy::ReplayVision` snapshots use selected player fog or player-union fog. `DiagnosticPolicy::ReplayObserverAnalysis` sends observer analysis to replay viewers. | `PersistencePolicy::NONE`; the original normal match may already have persisted a replay artifact. | `StartPayloadPolicy::ReplayViewer` with replay metadata, `capabilities.roomTime` speed/pause/seek/timeline, `capabilities.visibility.replayVision`, and `AffordancePolicy::ReplayViewer`. Client mounts `ReplayViewer`, replay speed/pause/seek, replay vision buttons, timeline, and branch button. | End a match into replay, change speed, pause/resume, seek, switch vision, request a branch. |
| Persisted replay room | `RoomMode::Replay` holds a persisted replay artifact. The room starts in lobby-shaped setup, prompts on join, then transitions to replay playback after confirmation. | Room ticker before the viewer starts; replay playback clock after `Phase::ReplayViewer` is active. | `MutationPolicy::None` before confirmation; replay viewer controls only after playback starts. | Lobby-shaped before confirmation; replay vision and observer analysis after confirmation. | Playback itself does not write match history. | Replay start payload is sent after confirmed join; client uses replay viewer affordances. | Open a match-history replay room, confirm the prompt, verify playback starts at replay tick zero. |
| Saved replay artifact | `RoomMode::ReplayArtifact` loads a saved self-play artifact from server target artifact directories. Confirmed join starts the shared replay runtime. | Room ticker before load/start; replay playback clock once the viewer is active. | `MutationPolicy::None` before confirmation; replay viewer controls only after playback starts. | Lobby-shaped before confirmation; replay vision and observer analysis after confirmation. | Playback itself does not write match history. | Replay start payload is sent after confirmed join; client uses replay viewer affordances. | Open a saved artifact replay, verify prompt, playback, vision selection, and timeline marks. |
| Replay branch staging | `RoomMode::ReplayBranch` holds a frozen replay branch seed and `BranchStagingState`. Joins attach to branch staging and are spectator-shaped until launch. | `ClockPolicy::BranchStaging`; no simulation ticks, but countdown can promote a ready branch to live. | Branch occupants claim and release original replay seats; branch host starts launch once every required seat is claimed. | Branch staging messages expose original seats and claims. No game snapshots or observer diagnostics. | Suppressed; staging does not write match history. | No game `start` payload while staging. Client mounts branch staging UI from `branchStaging` messages. | Claim/release seats, verify exclusivity, start only when all original human seats are claimed. |
| Replay branch live | Branch staging promotes to a cloned live `Game` from the replay keyframe. Connected ids are mapped to original replay seat ids. | `ClockPolicy::LiveMatch`. | Claimed occupants issue normal gameplay commands as their original replay seats through `MutationPolicy::BranchLiveSeatAliasGameplay`; unclaimed/extra viewers are spectators. | `VisibilityPolicy::LiveFog`; active branch seats receive player-fog snapshots for the original seats, and spectators receive active branch-seat union vision. | `PersistencePolicy::REPLAY_BRANCH_LIVE`; branch live matches keep transient post-match replay capture but suppress public match history and durable match-history replay artifacts. | `StartPayloadPolicy::ReplayBranchLive`; `AffordancePolicy::BranchLive`; claimed seats get prediction-enabled starts with mapped `playerId` and gameplay command capability, spectators get read-only spectator starts. | Launch a branch, command as a claimed original seat, verify spectator union vision and no public history write. |
| Dev scenario | `RoomMode::DevScenario` starts a purpose-built dev `Game` through `start_dev_session`; all connections are watchers. | `ClockPolicy::DevWatch`; `setRoomTimeSpeed(0)` pauses, positive `setRoomTimeSpeed` resumes/clamps, `stepRoomTime` advances one tick while paused. | Dev-watch controls mutate the dev driver clock and scenario tick progression through `MutationPolicy::DevScenarioDriver`. Viewer gameplay commands are ignored. | `VisibilityPolicy::DevFullWorld`; dev setup enables full projected movement debug paths represented by `DiagnosticPolicy::DEV_MOVEMENT_PATHS`. Client URL-local `devWatch` state still handles launch routing and no-fog display, but room-time controls and waypoint diagnostics come from start-payload capability metadata. | `PersistencePolicy::SUPPRESSED`; dev-watch end flow does not capture replay artifacts for history or post-match playback. | Normal `start` payload with `spectator: true`, `capabilities.roomTime` speed/pause/step, diagnostic capabilities from projection policy, plus `AffordancePolicy::DevWatch` and URL-local dev-watch routing affordances. | Open `/dev/scenario`, pause with speed 0, step one tick, resume, confirm path diagnostics and no-fog view. |
| Lab operator | `RoomMode::Lab` starts a real room-owned `Game` with lab metadata. First joiner becomes operator; lab session tracks dirty state, vision, and operation log. | `ClockPolicy::LiveMatch`; no special lab time control in this baseline. | Operator sends privileged lab envelopes represented by `MutationPolicy::LabPrivilegedOps`: setup mutations call public `Game` lab APIs, `issueCommandAs` routes gameplay commands as a selected player, `setVision` changes lab projection, import/export reads or restores scenario state, and operation logging stays room-local. | `VisibilityPolicy::LabFullWorld` by default; team/team-union lab vision can switch snapshots to `snapshot_for_spectator`. Observer analysis is not mounted. | `PersistencePolicy::LAB_ROOM_LOCAL`; `ExportPolicy::LAB_SCENARIO` allows scenario import/export through lab op results but is not durable public storage. | `StartPayloadPolicy::Lab` with `lab` metadata role `operator`; `AffordancePolicy::Lab`; start capabilities keep normal gameplay command surfaces read-only while lab issue-as remains lab-owned. Client mounts `LabPanel` and lab control policy. | Open `/lab`, spawn/move/delete or edit resources, switch lab vision, export a scenario, issue a command as one player. |
| Lab read-only viewer | Additional lab joiners attach to the existing lab session as read-only viewers and receive the same running lab `Game`. | Same live match clock. | Read-only viewers cannot send lab requests; normal gameplay command surfaces route through lab policy and reject non-operator issue-as. | Same lab projection as the operator, including full-world or selected-team lab vision. Observer analysis is not mounted. | No separate persistence/export authority. | Lab start payload has role `readOnly`; client gets lab state but no operator authority. | Join a lab from a second client, confirm the viewer sees lab state/snapshots but lab mutation requests are rejected. |

## Mode-Shaped Leaks Inventory

| Leak | Current shape | Remove or replace in |
| --- | --- | --- |
| Replay-named time commands | Replaced by neutral `setRoomTimeSpeed`, `seekRoomTime`, `seekRoomTimeTo`, and `stepRoomTime`. | Done in Phase 3 |
| Replay-named time state | Replaced by neutral `roomTimeState` for replay playback and dev-watch pause/speed state. | Done in Phase 3 |
| URL-local `devWatch.kind` controls | URL-local dev-watch state remains only for launch routing and no-fog dev display; shared room-time controls come from `startPayload.capabilities`. | Intentional setup/routing edge |
| Start payload `debugMode` | Replaced by recipient-scoped `diagnostics` metadata; `GameState` no longer reads `startInfo.debugMode`, and dev waypoint diagnostics no longer come from `devWatch.kind`. | Done in Phase 4 |
| `DebugHuman` diagnostic coupling | Movement debug paths are requested through neutral `SnapshotOptions` chosen by room projection policy, not through `Game.debug_path_overlays` or `StartingLoadout::DebugHuman`. | Done in Phase 4 |
| Projection diagnostics tied to vision | `ProjectionPolicy` now carries independent observer-analysis and movement-path diagnostic axes while still pairing them with explicit visibility choices for each recipient. | Done in Phase 4 |
| Replay identity in client affordances | `ReplayControls` receives parsed capability metadata for room-time and replay-vision controls. The replay branch button remains product-owned replay UI. | Done in Phase 5/6 |
| Replay branch live joining | `SessionPolicy` now reports `JoinPolicy::BranchLiveAttach` for live replay branches, while `RoomTask` uses a shared branch-room join helper for the current attach behavior. | Done in Phase 2 |
| Match-history helper by mode | `SessionPolicy` now splits match-history eligibility, transient post-match replay capture, durable replay-artifact attachment, and lab operation logging. `RoomTask` still combines that with participant count, automation filters, and DB gating. | Done in Phase 6 |
| Lab mutation hidden behind authority | `SessionPolicy` now exposes `MutationPolicy::LabPrivilegedOps`, `ExportPolicy::LAB_SCENARIO`, `PersistencePolicy::LAB_ROOM_LOCAL`, and `AffordancePolicy::Lab`; per-operator validation and individual lab op routing intentionally remain in `RoomTask::on_lab_request`. | Done in Phase 6 |
| Lab projection uses replay helper for team vision | Lab selected-team projection still uses the union snapshot helper, but diagnostics stay disabled unless `ProjectionPolicy` explicitly enables them and snapshot calls are guarded through `projection.rs`. | Intentional projection helper reuse |
| Observer overlay mount by product booleans | Client observer analysis mounts from explicit diagnostic metadata in parsed room capabilities, not replay/spectator booleans. | Done in Phase 5 |

## Final Guardrails

- `scripts/check-lobby-architecture.mjs` requires generic launch, live tick, projection, and fanout
  helpers to consume policy/capability metadata instead of `RoomMode` or `SessionMode` shortcuts.
- `scripts/check-lobby-architecture.mjs` requires end-match replay capture and match-history replay
  artifact attachment to stay gated by named persistence-policy helpers.
- `scripts/check-client-architecture.mjs` requires `room_capabilities.js` to parse
  `startPayload.capabilities` and `startPayload.diagnostics` instead of replay/dev/debug identity.
- `tests/protocol_parity.mjs` checks that the start-payload capability DTOs are documented and
  mirrored by the client parser.
