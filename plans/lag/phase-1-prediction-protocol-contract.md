# Phase 1 - Prediction Protocol Contract

## Objective

Add the wire-level contract needed for client prediction and server reconciliation. This phase
should not enable local simulation yet; it should make every player command and every authoritative
snapshot correlate cleanly.

## Protocol Work

- Add a monotonically increasing client-local sequence id to gameplay commands:
  - `clientSeq: u32`
  - generated by the browser per match, per connection
  - attached only to `command` messages, not lobby/control messages
  - mandatory for live player `command` messages; this project is pre-alpha and does not preserve
    compatibility with old live clients
  - prefer the command envelope shape `Command { clientSeq: u32, cmd }` over adding
    `clientSeq` to every `Command` variant, so replay/sim command DTOs do not inherit
    prediction-only transport metadata unless a later phase deliberately changes that contract
- Add acknowledgement metadata to every live gameplay snapshot:
  - `lastSimConsumedClientSeq: u32` for the highest contiguous local command sequence consumed by
    the authoritative simulation tick stream for this recipient
  - `lastSimConsumedClientTick?: u32` if useful for debugging/replay
  - `predictionVersion: u32` or equivalent protocol version marker
- Carry acknowledgement metadata through both snapshot encodings:
  - semantic/object `Snapshot`
  - compact live snapshot transport (`COMPACT_SNAPSHOT_VERSION`,
    `serialize_compact_snapshot`, and `decodeCompactSnapshot`)
- Define commands without `clientSeq` as invalid for live player control. The server may ignore
  them or emit a local protocol error, but it must not execute them in live matches.
- Define wraparound behavior. Prefer rejecting wraparound within a match by using a large enough
  integer range and resetting on match start.
- Document two acknowledgement milestones:
  - socket/room receipt: the server received a sequenced command and queued it for the room; useful
    for diagnostics only
  - sim consumption: the authoritative tick stream consumed the command; this is the only
    acknowledgement that reconciliation uses to drop pending predicted commands
- Document that sim-consumption acknowledgement does not mean "the command succeeded." A command can
  be acknowledged and still have no effect because the server rejected it during authoritative
  validation.

## Server Work

- Preserve arrival-order command behavior.
- Track socket/room receipt internally as optional diagnostics, but do not expose it as the
  reconciliation ACK.
- Store the highest contiguous sim-consumed `clientSeq` per connected active player.
- Include sim-consumption acknowledgement metadata in that player's fog-filtered snapshots.
- Keep `clientSeq` tracking in the room/connection layer so the `Game` API remains a pure
  authoritative command seam.
- Record `clientSeq` in auxiliary test logs, not replay command logs, unless a later phase
  deliberately changes the sim/replay contract and updates `docs/design/server-sim.md` plus replay
  artifact contracts in the same change.
- Keep spectators and replay viewers out of prediction acknowledgement unless a later phase adds
  replay-specific debugging fields.

## Client Work

- Add command sequence allocation in the transport/input boundary.
- Ensure every command path uses the same sequence allocation:
  - viewport right-click
  - minimap right-click
  - command card abilities
  - build placement
  - train/research/cancel
  - stop/setup/teardown/rally commands
- Decode mandatory snapshot acknowledgement fields for live player matches.

## Verification

- Rust protocol serde tests for sequenced command messages and rejection/ignore behavior for live
  commands missing `clientSeq`.
- JS protocol decode/build tests for every command builder that can issue gameplay commands.
- Compact snapshot transport test proving acknowledgement metadata survives the actual live
  serializer and JS decoder, with `COMPACT_SNAPSHOT_VERSION` bumped when the compact shape changes.
- Integration test with artificial command bursts asserting snapshots eventually acknowledge the
  highest sim-consumed sequence.
- Regression test that live commands without `clientSeq` do not execute.
- Auxiliary log/test hook proving acknowledged commands are recorded in the same order they are
  consumed, without changing deterministic replay command logs.
- Protocol documentation update in `docs/design/protocol.md` in the same change.

## Player-Facing Outcome

No gameplay change yet. This phase creates the acknowledgement contract that makes later immediate
local response correctable instead of guesswork.
